# 05 编码规范

## 1. 目标

统一编码、设计和日志要求，确保系统可读、可改、可评审。

## 2. 编码规范

- 一个函数只承担一个主要职责。
- 失败路径必须显式可见。
- 避免吞异常和模糊命名。
- 避免跨层直接依赖实现细节。
- adapter 不承载核心业务规则。
- 接入层优先复用统一 helper，减少重复的 proto 解析、错误回包和响应封装模板。
- 只有在确实需要扩展默认行为时才覆盖基类 hook，避免保留无意义的样板 override。

## 3. 设计规范

- 新增逻辑先判断落在哪一层。
- 协议模型、上下文模型、领域模型不要混写。
- `gateway_server_app` 只做接入和转发，不做业务决策。
- `LoginService`、`PlayerService`、`DungeonService` 作为应用服务承载编排，业务规则优先沉淀到应用层和领域层。
- 应用服务内部优先拆成固定步骤，避免把校验、仓储访问、规则判断和结果组装揉成一个长函数。
- 成功结果和失败结果优先集中构造，避免在主流程里散落重复的聚合初始化。

## 4. 日志规范

关键请求日志至少记录：

- `trace_id`
- `request_id`
- `session_id`
- `player_id`
- `connection_id`
- `message_id`
- `executor_label`
- `executor_shard`

必须为以下事件提供清晰日志：

- 请求失败
- 上游超时
- 队列满
- shutdown drain 超时
- session restore 失败

## 5. 评审规范

评审优先关注：

- 分层是否正确
- 模型是否混乱
- 失败路径是否完整
- 并发语义是否正确
- 日志是否足够定位问题
