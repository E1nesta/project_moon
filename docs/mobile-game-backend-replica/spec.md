# 功能规范（spec）

## 基本信息
- 生成日期：2026-03-28
- 需求主题：复现一套面向手游的 C++/Linux/MySQL/Redis 后端系统，覆盖账号鉴权、角色数据管理、副本进入与结算、日志排查与联调能力

## 背景与目标
- 背景：基于既有项目经历，抽取手游服务端最核心的业务闭环，复现一套可运行、可演示、可继续扩展的后端系统。复现对象聚焦账号登录、角色加载、单人副本进入与结算三条主链路，避免一开始引入公会、聊天、支付、跨服等复杂模块。
- 目标：在单机或小规模 Linux 环境中完成网关、登录服、游戏服、MySQL、Redis 的协同运行，并让客户端或测试脚本可以完整跑通“登录 -> 加载角色 -> 进入副本 -> 提交结算 -> 发放奖励 -> 持久化数据”的流程。

## 范围
- 包含范围：
  - 账号鉴权：账号注册样例、密码校验或测试令牌校验、登录态生成、重复登录处理。
  - 角色数据：角色基础信息、资源属性、背包、体力、副本进度、在线状态加载与保存。
  - 副本系统：副本进入条件判断、体力扣减、挑战记录创建、结算校验、首通奖励、普通掉落、重复领奖拦截。
  - 基础支撑：配置表加载、统一日志、错误码、GM 或测试指令、接口联调脚本、关键流程回归用例。
  - 存储层：MySQL 持久化、Redis 缓存与分布式锁、玩家会话缓存。
- 不包含范围：
  - 支付、邮件、排行榜、公会、聊天、匹配、跨服、战斗服拆分。
  - 完整反作弊平台，只保留结算合法性校验和重放保护。
  - 多角色复杂切换、复杂道具经济系统、后台运营平台。

## 核心需求
1. 功能行为：
   - 客户端提交账号凭证后，系统完成鉴权并返回登录态、角色列表或默认角色信息。
   - 玩家进入游戏后，系统从 MySQL 读取角色基础数据，并结合 Redis 中的在线状态、缓存快照恢复会话。
   - 玩家请求进入副本时，系统校验等级、体力、章节开放条件、副本配置与是否存在未结算挑战记录。
   - 玩家副本结算时，系统依据挑战记录、结算上下文和配置表完成奖励计算、首通判定、资源发放与落库。
2. 关键规则：
   - 一个账号同一时刻只允许一个有效在线会话；重复登录时以最新登录态为准，旧会话失效。
   - 角色加载采用“Redis 快速读取 + MySQL 最终一致”的策略，关键写入以 MySQL 事务为准。
   - 体力在进入副本成功时先扣减，不在结算时补扣；若创建挑战记录失败，则回滚本次扣减。
   - 每次挑战生成唯一 `battle_id`，结算必须带上对应 `battle_id`，且同一 `battle_id` 只允许成功结算一次。
   - 首通奖励按“角色 + 副本”维度判定，仅首次通关时发放；普通掉落按配置表与结算结果计算。
   - 重复领奖通过 Redis 锁与 MySQL 唯一约束双重保证，防止并发重复结算。
3. 异常处理：
   - 无效令牌、角色不存在、配置缺失、Redis 不可用、MySQL 写入失败时返回明确错误码并记录日志。
   - 体力不足、未解锁副本、挑战记录缺失、结算超时、重复结算、伪造关卡或伪造星级时拒绝结算。
   - 资源发放异常时必须确保事务整体回滚，避免出现“扣了体力但没发奖励”或“发了奖励但没落进度”的脏数据。

## 验收标准
1. 给定合法账号和角色数据，系统可在测试环境完成登录、角色加载和在线状态建立。
2. 给定满足条件的角色，系统可成功进入副本并正确扣减体力、创建挑战记录。
3. 给定合法 `battle_id` 和结算参数，系统可正确发放普通掉落、首通奖励并更新副本进度。
4. 给定重复结算请求或重复领奖请求，系统稳定返回拦截结果，且数据库不出现重复奖励记录。
5. 给定非法副本编号、非法星级、越权结算或缺失挑战记录，系统拒绝结算并输出可排查日志。
6. 关闭服务前触发角色保存后，重新登录可恢复正确的资源、背包和副本进度数据。

## 依赖与约束
- 外部依赖：
  - Linux 开发环境，建议 Ubuntu 22.04 或同级发行版。
  - MySQL 8.x 用于持久化，Redis 7.x 用于缓存、锁和会话。
  - CMake、gcc/clang、可选 protobuf 或 JSON 序列化库。
  - 配置表来源可先用 CSV/JSON，启动时加载到内存。
- 技术约束：
  - 首版以单区单服、单进程网关 + 登录服 + 游戏服为主，优先保证业务闭环和代码结构清晰。
  - 服务端采用 C++17，使用事件驱动网络模型，避免过早引入微服务复杂度。
  - 关键写路径必须具备幂等保护、事务保障和结构化日志，便于联调和问题复现。
  - 所有关键错误码和日志字段需要统一命名，便于后续接入监控与自动化测试。
