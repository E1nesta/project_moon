# 技术选型说明

## 1. 选型目标

本项目定位为：

- 社招向手游后端复现项目
- 目标读者和实现者以新手为主
- 强调能讲清楚、能跑起来、能展示业务闭环

因此选型原则不是“最新最全”，而是“成熟、稳定、低复杂度、易解释”。

## 2. 最终选型

- 语言标准：`C++17`
- 目标环境：`Linux`
- 构建工具：`CMake`
- 网络通信：`TCP + 事件驱动模型`
- 协议定义：`protobuf`
- 持久化：`MySQL 8`
- 缓存与锁：`Redis 7`
- 日志：结构化日志，后续可接 `spdlog`
- 本地开发：`Docker Compose`

## 3. 为什么这样选

### C++17

- 足够成熟，资料多，面试表达更自然。
- 对新手更友好，不需要为了语言特性额外增加理解成本。
- 足以支撑服务端常见需求，如网络、对象管理、配置加载、数据访问封装。

### MySQL

- 角色数据、资源、背包、副本进度、奖励流水都很适合关系型建模。
- 事务能力强，适合处理副本结算这类“一次操作改多张表”的场景。

### Redis

- 适合会话缓存、玩家锁、角色快照、挑战上下文。
- 读写性能高，适合作为 MySQL 之前的一层缓存和并发控制组件。

### protobuf

- 适合客户端和服务端之间定义稳定消息结构。
- 后续若需要服间通信，也可以继续复用，不必重复造协议格式。

## 4. 当前不引入的技术

首版明确不引入以下复杂度：

- 不上 `gRPC`
- 不上 `bRPC`
- 不上消息队列
- 不上 ORM
- 不做复杂微服务治理
- 不做多区多服分布式架构

原因不是这些技术不好，而是当前阶段没有必要。

项目最重要的展示点是：

- 账号鉴权
- 角色加载与保存
- 副本进入条件判断
- 体力扣减
- 奖励发放
- 首通判定
- 重复领奖校验
- 非法结算拦截

这些能力用 `C++17 + MySQL + Redis + protobuf` 已经足够完成。

## 5. RPC 的态度

当前阶段不引入 RPC。

原因：

- 客户端到服务端更适合长连接和自定义业务协议。
- 当前系统规模小，服务之间完全可以先通过简单模块调用或轻量消息转发解决。
- 过早引入 RPC 会抬高工程复杂度和学习成本。

后续只有在以下情况再考虑：

- 登录服、游戏服、活动服真正拆成独立进程并长期演进
- 出现明显的跨语言协作需求
- 内部服务调用链复杂到必须统一治理

如果未来需要补 RPC：

- 纯 C++ 内部服务优先考虑 `bRPC`
- 跨语言或标准化生态需求再考虑 `gRPC`

但这属于二期演进，不属于首版必需品。

## 6. 一句话结论

这套项目当前最合适的技术路线就是：

`C++17 + Linux + MySQL + Redis + protobuf + CMake`

保持简单，把业务闭环做扎实，比继续堆技术名词更重要。
