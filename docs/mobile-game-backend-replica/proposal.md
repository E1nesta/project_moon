# 变更提案（proposal）

## 基本信息
- 生成日期：2026-03-28
- 变更主题：复现一套面向手游的 C++/Linux/MySQL/Redis 后端系统，覆盖账号鉴权、角色数据管理、副本进入与结算、日志排查与联调能力

## 变更动机
- 业务动机：
  - 将项目经历中的核心业务链路沉淀为可展示、可讲解、可二次开发的作品。
  - 用一个完整闭环证明自己具备手游服务端常见模块的设计、编码、联调和问题排查能力。
  - 为后续扩展商城、邮件、任务、排行榜等玩法提供稳定基座。
- 技术动机：
  - 简历中的业务描述覆盖面较广，但缺少可直接演示的系统实现，需要抽象出最小可运行系统。
  - 账号、角色和副本结算涉及缓存、一致性、幂等、事务、异常补偿，是非常适合展示后端工程能力的组合。
  - 通过复现过程，可以沉淀统一网络层、数据访问层、配置系统和日志体系，降低后续继续扩展的成本。

## 方案概述
- 方案摘要：
  - 采用“三层职责”方案：网关负责连接与路由，登录服负责账号鉴权与会话签发，游戏服负责角色与副本业务。
  - MySQL 作为最终持久化存储，Redis 承担在线会话、角色快照、玩家锁和挑战上下文缓存。
  - 副本结算以 `battle_id` 为幂等主键，结合 Redis 锁和 MySQL 唯一索引，保证重复请求不会重复发奖。
  - 首版只实现单人 PVE 副本，不拆分独立战斗服，所有结算合法性在游戏服中完成。
  - 通过结构化日志、测试脚本和 GM 指令支撑联调、回归和故障排查。
- 关键设计：
  - 服务拆分：`gateway`、`login_server`、`game_server`、`common` 公共库。
  - 领域模块：账号鉴权、玩家会话、角色档案、背包资源、副本挑战、奖励结算。
  - 数据模型：账号表、角色主表、角色资源表、背包表、副本进度表、奖励流水表、挑战结算表。
  - 一致性策略：进入副本扣体力与创建挑战记录放在同一事务中；结算发奖、进度更新和流水记录放在同一事务中。
  - 可观测性：为登录、加载、进副本、结算、回滚五类关键行为增加统一 trace 字段。

## 影响评估
- 影响系统：
  - 新增网关、登录服、游戏服三个可独立启动的服务进程。
  - 新增 MySQL 表结构、Redis Key 规范、配置表解析模块和测试脚本。
  - 新增服务端基础库，包括网络封包、协议分发、错误码、日志、数据库与缓存访问封装。
- 影响用户：
  - 对演示用户或测试客户端来说，可感知到完整的登录、选角、挑战副本、领取奖励闭环。
  - 首版不会提供完整运营玩法，因此玩法深度较浅，但主路径清晰、便于验证。
- 影响数据：
  - 需要初始化账号和角色样例数据。
  - 需要建立副本配置、掉落配置和首通奖励配置。
  - 若后续扩展玩法，可沿用当前账号、角色、道具和流水模型继续演进。

## 风险与回滚
- 主要风险：
  - 进入副本和结算流程涉及多处状态变更，容易出现缓存与数据库不一致。
  - 如果过早把模块拆得过细，会增加联调成本和首版复杂度。
  - 非法结算判定规则如果设计过松，容易让系统看起来只有“发奖”而没有“校验”。
- 缓解措施：
  - 先做单服闭环，再考虑拆分更多服务或引入异步队列。
  - 所有关键写路径统一经过玩家锁，确保同一角色请求串行化。
  - 对副本结算增加挑战记录校验、时间窗口校验、配置校验和重复结算校验。
  - 先用测试脚本覆盖边界场景，包括断线重连、重复结算、资源不足、配置缺失等情况。
- 回滚策略：
  - 每个阶段单独打 Tag，若新功能不稳定，可回退到上一个可运行里程碑。
  - 结算模块异常时，可临时关闭结算入口，只保留登录和角色加载功能继续验证。
  - 数据层面保留初始化 SQL 和清库脚本，便于快速重建测试环境。

## 验证计划
1. 功能验证：
   - 登录成功/失败、重复登录挤下线、角色加载、角色保存。
   - 副本进入条件通过/失败、体力扣减、挑战记录创建。
   - 副本结算成功、首通奖励触发、重复结算拦截、非法结算拒绝。
2. 回归验证：
   - Redis 缓存失效时是否可以从 MySQL 恢复角色数据。
   - 服务重启后是否能恢复会话与角色持久化结果。
   - 高并发下同一角色重复请求是否被正确串行化。
3. 发布验证：
   - 启动脚本是否可一键拉起依赖和服务。
   - 日志中是否能完整追踪一次登录和一次副本结算链路。
   - 关键指标如登录成功率、结算成功率、重复领奖拦截次数是否可观测。
